 Listo. El lado cliente ya queda preparado para /authusercredentials. Qué hace
  ahora el botón Continuar (en el paso de credenciales):

  1. Valida el formato local de usuario/contraseña (validarAuth) — atrapa typos
  antes de molestar al dispositivo.
  2. Bloquea el botón y muestra "Verificando…" para evitar dobles envíos.
  3. POST /authusercredentials con { user, pass } (mismo formato y headers que
  /newparams y /usercredentials, para consistencia).
  4. Según la respuesta:
    - res.ok && json.status → credenciales correctas: desbloquea el modo edición
  (setEstado("edit") + transición).
    - Si no → alert con json.message del dispositivo (o "Usuario o contraseña
  incorrectos."), sin destruir el formulario, para que reintenten.
    - Error de red → alert("Error de comunicación con el dispositivo").
  5. finally restaura el botón ("Continuar", habilitado) en cualquier caso.

  Contrato que espera del backend (para que lo implementes en el dispositivo):
  - Request: POST /authusercredentials, body JSON { "user": "...", "pass": "..."
  }.
  - Response: JSON con { "status": true|false, "message": "..." }. status: true
  (y HTTP 2xx) = credenciales válidas; cualquier otra cosa = rechazo y se
  muestra message.
  
  Notas:
  - El guardado final (/newparams) sigue enviando user+pass junto con los
  parámetros, así que el dispositivo puede re-validar al guardar como defensa en
  profundidad (no confía solo en el paso previo del cliente).
  - Esto cierra la brecha que te señalé antes: ya no se desbloquea la edición
  solo por formato válido, sino por verificación real contra el dispositivo.
  
  JS verificado sin errores de sintaxis. Si tu endpoint usa otras llaves (p. ej.
  password en vez de pass, o un campo distinto a status), dímelo y lo ajusto.
